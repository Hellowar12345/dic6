## Context

台灣都市停車問題日益嚴峻，800 萬輛汽車中有大量集中於六都市中心。現行停車 App（如停車大聲公、台北好停車）僅提供即時剩餘車位顯示，存在「行車時間差」問題——駕駛看到有空位出發後，抵達時車位已被佔滿。

政府 TDX 平台提供即時與歷史停車資料，氣象局提供天氣 API，但尚無系統性地將這些多源資料整合並透過深度學習模型進行前瞻性預測。

本系統目標是建立輕量化、可即時部署的四層式 DD-LSTM（Data-Driven LSTM）端對端預測管線，初期以台北市公有路外停車場為驗證場域。

## Goals / Non-Goals

**Goals:**
- 建立可精準預測未來 30/60 分鐘停車位可用率的 DD-LSTM 模型
- 整合多源外部特徵（天氣、假日、時段週期）提升預測精度
- 提供直覺的 Web Dashboard 即時呈現預測結果與警示
- 模型預測準確率須優於 ARIMA、Random Forest、單層 LSTM 等 baseline
- 系統架構需具備全台擴充性（支援新增停車場無需修改核心模型）

**Non-Goals:**
- 不實作導航系統或路線規劃功能
- 不建置原生行動 App（僅 Web Dashboard）
- 不處理私人停車場資料（僅公有路外停車場）
- 不實作即時車牌辨識或 IoT 感測器整合
- 不實作計費或支付功能

## Decisions

### 1. 模型架構：DD-LSTM（雙層 LSTM + Dropout）
**選擇**: 兩層 LSTM（各 64 units）搭配 Dropout 0.2，加上 Dense 輸出層

**替代方案考量**:
- **單層 LSTM**: 結構簡單但對複雜時序模式的擬合能力不足
- **Transformer**: 雖在長序列上表現優異，但對即時推論延遲較高、部署較重
- **GRU**: 參數較少但在此類多特徵時序預測上效果略遜 LSTM

**理由**: 雙層 LSTM 在保持合理推論速度的同時，能更好捕捉停車場佔用率的多層時序特徵。Dropout 0.2 有效防止過擬合。

### 2. 滑動視窗大小：Window Size = 24
**選擇**: 以 24 個時間步（每步 5 分鐘 = 2 小時歷史）作為輸入序列長度

**替代方案考量**:
- **Window = 12（1 小時）**: 歷史資訊不足以捕捉通勤高峰完整模式
- **Window = 48（4 小時）**: 增加計算成本而邊際收益遞減
- **Window = 288（24 小時）**: 序列過長導致梯度消失風險

**理由**: 2 小時涵蓋一個完整通勤尖峰週期，資訊量與計算效率的最佳平衡

### 3. 正規化方式：Min-Max Scaling
**選擇**: Min-Max Scaling 至 [0, 1] 範圍

**替代方案考量**:
- **Z-score 標準化**: 適合常態分布數據，但佔用率數據右偏（高佔用率常聚集）
- **Robust Scaling**: 對離群值穩健但增加複雜度

**理由**: 佔用率本身為 [0%, 100%] 有界值，Min-Max Scaling 最自然且便於反正規化

### 4. 資料擷取頻率：每 5 分鐘
**選擇**: Cron Job 每 5 分鐘透過 TDX REST API 擷取即時資料

**理由**: TDX API 更新頻率約為 5 分鐘，過高請求頻率無額外資訊增益且可能觸發 API 限流

### 5. 前端技術棧：Vanilla HTML/CSS/JS + Chart.js
**選擇**: 不使用前端框架，直接以原生技術搭配 Chart.js 圖表庫

**替代方案考量**:
- **React/Vue + D3.js**: 功能強大但增加部署複雜度
- **Grafana Dashboard**: 現成方案但客製化受限

**理由**: 與 PRD 的「輕量化部署」原則一致，降低依賴與維運成本

### 6. 時序資料庫：InfluxDB
**選擇**: 以 InfluxDB 儲存停車場歷史佔用率時序資料

**理由**: 針對時序資料優化的寫入/查詢效能，原生支援 downsampling 與 retention policy

## Risks / Trade-offs

| 風險 | 影響 | 緩解策略 |
|------|------|----------|
| TDX API 不穩定或停機 | 即時資料中斷導致預測失準 | 實作 fallback 機制（使用最近一次有效資料 + 歷史同期平均值） |
| 天氣 API 預報錯誤 | 天氣特徵失準影響模型輸出 | 天氣特徵作為輔助變數，權重較低；設計降級模式（僅用時序特徵預測） |
| 突發事件（如大型活動、道路封閉） | 模型無法預見非週期性需求激增 | 預留手動覆蓋機制；後續版本整合即時事件 API |
| 模型冷啟動（新增停車場） | 新場域缺乏歷史資料 | 使用相似區域的遷移學習 + 規則基礎的初始預測 |
| 前端即時性需求 | Dashboard 資料延遲影響使用者信任 | WebSocket 或 SSE 推送更新；顯示「最後更新時間」透明化 |
| 季節性模式飄移 | 長期未重訓練的模型精度下降 | 實作線上學習或排程定期重訓練（每月） |
